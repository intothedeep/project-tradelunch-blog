## Product Identity

Personal portfolio + fintech dashboard for Taek Lim (GitHub: tradelunch).
The site serves two distinct purposes under one Next.js 16 monorepo:

1. **Blog / portfolio** — curated technical posts fetched from a live backend (`article_server`), rendered with full Markdown/LaTeX pipeline, infinite scroll.
2. **Financial markets dashboard** — live market snapshot for personal reference. Simple, readable, no customization needed.

Domain: `my.prettylog.com`. Navigation exposes: About, Blog, Resume, (Dashboard not yet linked in nav).

---

## Tech Stack (locked)

- Next.js 16, App Router, TypeScript strict
- Turborepo monorepo: `apps/dashboard_client_web`, `apps/article_server`, `packages/*`
- Tailwind CSS v4, Radix UI, shadcn/ui pattern, lucide-react
- Jotai, TanStack Query v5, next-intl, recharts

### Phase 2-simple — Financial Dashboard MVP (ACTIVE)

>

Chart library swapped from `recharts` to `lightweight-charts` after user requested OHLC candlesticks and 90-day history. nof1.ai/models/31 was reviewed as a visual reference but its editorial/parchment aesthetic was rejected in favor of a TradingView dark-terminal look. >
> Updated 2026-05-07 (later): User provided a TradingView screenshot as the visual target. Variant C v3 elevates the chart to "TradingView Essential Clone": MA overlays (5/20/50), volume histogram, current-price axis label, top ticker bar, bottom timescale buttons (display-only), watchlist redesigned with grouped sections + symbol icons. Skipped: drawing tools, indicators dropdown, alerts, bottom detail panel, real timeframe switching. These remain non-goals for MVP.

**Goal:** Display a live market snapshot across five categories in a clean, readable page — no customization, no drag/resize.

`FinancialGridLayout.client.tsx` uses `react-grid-layout` with ~230 lines of orchestration
code for a problem this product does not have (read-only snapshot, no drag/resize needed).
Replacing with a `grid-cols-*` Tailwind CSS grid reduces the layout component to ~30-50 lines.
Drop `react-grid-layout`.

#### Resolved Product Decisions

1. **Stock tickers (10):**
   - US (9): GOOGL, TSLA, AAPL, AMZN, META, SMR, TQQQ, SOXL, WMT
   - KRX (1): NCSOFT — ticker `036570`; Twelve Data suffix likely `036570.KS` — architect must verify this before implementing the fetcher.

2. **Refresh cadence by category:**
   - FX: 60s
   - Crypto: 30s
   - Indices (US): 60s during market hours; longer off-hours acceptable
   - Indices (KRX): once per market day (daily-close only via KRX OpenAPI)
   - Interest Rates: 24h
   - Stocks: 60s during market hours of the listing exchange
   - These are baseline targets; architect may adjust based on API rate limits.

3. **Display style — decided in Cycle 2:**
   - Variant A: card-based grid (one card per item — name / value / change); preview at `/dashboard/preview/cards`
   - Variant B: table-based per category (one table per category section); preview at `/dashboard/preview/table`
   - Variant C: chart on left (~60% width desktop) + table on right (~40%); clicking a table row updates the chart with that item's price history; **chart library: `lightweight-charts` v5+ (TradingView's open-source candlestick library)** — swapped from `recharts` on 2026-05-07 after user requested OHLC candles + 90-day history; preview at `/dashboard/preview/chart`
   - All three variants are built as preview pages using static mock data. User picks the winner in Cycle 2 review.

4. **Change indicator:** 1-day window, delta vs prior close. Format: signed percentage with up/down arrow, green = up, red = down.

5. **Data architecture — internal backend, not external APIs at request time** (decided 2026-05-06):
   - The Next.js app **does NOT** call Twelve Data, KRX OpenAPI, or any external market-data provider directly at request time.
   - External API ingestion is the responsibility of the user's own backend (`dashboard_server` or equivalent). The backend persists snapshots and price history into the user's database on a schedule that matches the refresh cadence in §2.
   - The Next.js app reads only from the user's backend — both for snapshot polling and for chart history on Variant C row clicks.
   - **Rationale:** rate limits and API keys stay server-side; click interactions in Variant C cannot drain the daily API credit budget; the Next.js client never holds external credentials.
   - **Development order:** Cycle 2 uses static mock data (no backend required). Cycle 3 swaps the mock fetchers for backend reads once the user has their DB schema and ingestion job in place. Until then, Cycle 3 tasks targeting "real data" remain pending on the backend availability.

#### MVP-done acceptance criteria (applies regardless of which display variant wins)

- All 22 items (12 base + 10 stocks) display a current value and 1-day change % with up/down color indicator
- Page renders in <2s on dev server with mock data
- No hydration errors (all date values passed as ISO strings, never `Date` objects in props)
- Zero `any` types in the new code paths
- `pnpm --filter dashboard_client_web check-types` exits 0
- No API keys hardcoded; all sourced from `.env`
- Loading skeleton visible during fetch; error state shows readable message (not raw error object)
- Dashboard reachable from main navigation (desktop + mobile)
- If Variant C wins: clicking a table row updates the chart in <100ms perceived latency with mock data.

#### Non-goals for MVP (explicit scope boundary)

- Drag-and-drop or resize of any panel
- Historical charts beyond a single line chart per selected item are non-goal; advanced features like indicators, multiple time-windows, drawing tools are out of scope.
- Watchlist management or user customization
- Alert / notification system
- Real-time WebSocket streaming
- Any historical range selector
