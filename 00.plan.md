## Architecture Decisions (resolved)

**`@repo/types` package (2026-06-24).** Lean package (no build step, raw TS, `main`/`types` →
`src/index.ts`) for API-boundary types shared front+back: `ETreeNodeType`, `TCategoryTreeNode`,
`TPostTreeNode`, `TCategoryTreeResponse`, `TTreeNode`, `TTreeNodeWithChildren`. Client adds
`transpilePackages: ['@repo/types']`. `@repo/markdown-parsing` later retired from the client
(TOC inlined to `utils/markdown/`).

**Env var convention (kcdd2).** SCREAMING_SNAKE_CASE, no prefix namespacing (`DATABASE_URL`,
not `SUPABASE_DATABASE_URL`). Key vars: `DATABASE_URL` (Supabase pooler 6543), `NODE_ENV`,
`SITE_DOMAIN`, `API_SITE_DOMAIN`, `CDN_ASSETS`, `SUPABASE_STORAGE_BUCKET` (default `blog.prettylog`).
**Renamed (2026-06-26):** `CDN_ASSET_POSTS`→`CDN_ASSETS` (server) /
`NEXT_PUBLIC_CDN_ASSET_POSTS`→`NEXT_PUBLIC_CDN_ASSETS` (client), default domain
`posts.prettylog.com`→`assets.prettylog.com` (commit 3100a9f); the image bucket is env-driven via
`SUPABASE_STORAGE_BUCKET` (default `blog.prettylog`, replaced hardcoded `blog-images`, commit 20dbf26).

**Data access.** Raw SQL kept verbatim. Sequelize execution layer replaced by `pg` Pool
(`max: 1`) → Supabase transaction pooler (port 6543). No supabase-js. The 591-line `posts.ts`
SQL is unchanged.

**Pagination cursor contract (2026-06-26).** Post ids are **Snowflake** (~3.29e17, generated by
blog_agent) and EXCEED JS `Number.MAX_SAFE_INTEGER` (~9.0e15). All keyset list endpoints
(`p.id < cursor`) therefore treat the cursor as an **opaque numeric string**, default the maximum
int8 `'9223372036854775807'`, bind it as a string, and return `nextCursor` as the full-precision id
string. `@repo/types` `TAdminPostListResponse.nextCursor` is `string | number | null`. See
Decisions Log #24.

**Backend topology.** `dashboard_server` stays in the monorepo at `apps/dashboard_server`.
Two Vercel projects, one repo. Vercel Root Directory = the app dir; install/build from repo root.

**Vercel tsconfig strict-narrowing constraint (2026-06-26).** `@vercel/node` type-checks the
serverless function import graph with a bundled `ts-node@10.9.1`, which **cannot resolve a tsconfig
`extends` that points at a workspace package** (`@repo/typescript-config`) under TypeScript ≥5.3 —
it silently drops inherited `strict`/`strictNullChecks`, collapsing discriminated-union narrowing
(`TS2339` on `result.reason` after `if(!result.ok)`). Fixed in ts-node 10.9.2 but Vercel pins
10.9.1 (ref: github.com/vercel/vercel/issues/12087). **Resolution:** `apps/dashboard_server/tsconfig.json`
is **self-contained** — it inlines the compiler options (target/module/strict/esModuleInterop/
skipLibCheck/composite/ignoreDeprecations) instead of extending `@repo/typescript-config`; and
`vercel.json` `buildCommand` is `pnpm --filter @repo/types build && pnpm --filter dashboard_server
check-types`, a deterministic strict gate via real `tsc`, independent of `@vercel/node` internals.
No application code changed; the `src/helpers/*` discriminated unions were correct. **Standing
constraint:** any tsconfig that `@vercel/node` reads MUST inline strict (no workspace `extends`);
only this one server tsconfig is self-contained, all other workspaces still
`extends @repo/typescript-config`.

**Keep `@repo/types` centralized (2026-06-26, architect-validated).** `@repo/types` stays the
single source of truth for the client↔server HTTP contract — do **NOT** duplicate types per-app
(duplication would cause silent drift). `@repo/types` was **not** the cause of the Vercel TS2339
deploy failure (the failing types were local to `dashboard_server/src/helpers/*`); the fix was the
self-contained server tsconfig above.

**blog_agent merge (2026-06-25).** The Python LangGraph publishing agent was merged into the
monorepo as `apps/blog_agent` (lift-and-shift, code logic unchanged). Turbo recognises it via a
thin `package.json` stub delegating to `uv run`; `start` is excluded from `dev`/`build`, only
`test`/`lint`/`check-types` are filterable. Schema SSOT is
`apps/dashboard_server/schema/tradelunch.schema.sql`; `apps/blog_agent/schema/…` is a byte-identical
copy guarded by `pnpm check:schema` (drift detection). See "blog_agent" under Phase A/B below.
Full ADR archived in `_docs/00.blog-agent-merge.arch.md`.

**blog_agent is a DIRECT-WRITE publisher (2026-06-26, architect-verified; first real publish landed
2026-06-26).** `apps/blog_agent` writes blog content **directly to the database and storage** — it
does **NOT** call the Express `/v1/api/posts` write API (that write path is the Clerk-authenticated
`/write` UI surface). The agent inserts/updates `posts`/`files`/`categories`/`tags`/join rows
directly into Postgres (SQLAlchemy/asyncpg over the Supabase pooler) and uploads images directly to
Supabase Storage bucket `blog.prettylog` (supabase-py). **Publisher identity is `user_id=1`** (the
single real `users` row — `taeklim`, Clerk-linked; the earlier `DEFAULT_USER_ID=2` never existed and
caused an FK violation — see Decisions Log #25); Clerk is not involved in the publish path. Express
only **READS** the agent's rows (the public blog feed + post detail). This is the intended division:
the agent is the single-author content producer; the Express server is the read API; the Clerk
`/write` UI is the separate multi-user authoring path (Phase D). Implication: publish→read
correctness depends on the agent writing the **same row shapes the server reads** (status default
`'public'`, `is_thumbnail`, `stored_uri` as an absolute CDN URL, tags/categories join shapes). The
first real publish succeeded end-to-end on 2026-06-26 (`java-spring-jdbc`, status='public', thumbnail
public URL, reads back via `GET /v1/api/posts`) — see Phase B and `00.tasks.md` B5.

Largest phase; do not begin until Phase B is stable (schema `0003` + the security filter below

provisioning — invite-only is no longer the active model (see "Provisioning UPDATE" below). The

**Central decision — auth enforcement (FINAL, two design agents reconciled):** Clerk verification
happens at the **Express layer**; the `pg` Pool + raw SQL stay. The Express server connects to
Postgres directly (service-equivalent credentials), so it **BYPASSES Supabase RLS by design**.
Therefore authorization is enforced in the **Express application/service layer, NOT in RLS**.
Every write derives `user_id` from the **verified Clerk token → mapped `users` row**, NEVER from
the request body. *(Rejected: a supabase-js + user-JWT rewrite of the 591-line raw SQL; rejected a
hybrid pg/supabase-js split.)*

**Integration method:** native Supabase **Third-Party Auth (TPA)** — register Clerk as a TPA
provider. For the Express path, even TPA is not required: Express verifies the Clerk JWT via
`@clerk/express` `verifyToken` against the Clerk JWKS using `CLERK_SECRET_KEY`. TPA/RLS is
**optional defense-in-depth for any FUTURE supabase-js read path only**. *(Deprecated and removed:
the "JWT template signed with the Supabase JWT secret" + `clerk_user_id()` SQL helper approach.)*

**Identity mapping:** Clerk owns identity (string `user_2ab…`). Supabase `users` keeps its integer
`id` PK; add a `clerk_user_id TEXT UNIQUE` bridge column. Request flow:
verify JWT → `sub` → `SELECT id,username,is_admin FROM users WHERE clerk_user_id=$1` → attach
`req.auth`. The blog_agent publisher identity is the real `users.id=1` row (corrected from the
non-existent `DEFAULT_USER_ID=2`, see Decisions Log #25) and is **never** a fallback for HTTP writes.

**Provisioning (ORIGINAL — superseded):** invite-only. Clerk `user.created` webhook →
`POST /v1/api/webhooks/clerk` (Svix-verified via `CLERK_WEBHOOK_SECRET`) → validate invite →
INSERT into `users` (idempotent, `ON CONFLICT (clerk_user_id) DO NOTHING`).

**Provisioning UPDATE (FINAL, 2026-06-26) — Svix webhook → lazy provisioning:** the `user.created`
webhook is replaced by **lazy provisioning** with `resolveAuth` as the single chokepoint. A valid
Clerk token with no `users` row creates a minimal row (`clerk_user_id` only, `username` NULL) via
race-safe `INSERT ... ON CONFLICT DO NOTHING` + UNION SELECT; `username` is later claimed at
`/onboarding`. The webhook is **soft-deleted** (`clerk.ts`→`x_clerk.ts`, `index`→`x_index.ts`),
`/api/webhooks` is **unmounted**, and the `express.raw()` body block is removed from `server.ts`.
- **Recap (provision-time fields):** at first authenticated hit ONLY `clerk_user_id` is stored;
  `email` and `username` are **NULL** (the token exposes only `sub`). `username` is claimed at
  onboarding; `email` is never backfilled. → admin bootstrap + any user lookup must key on
  `username` or `clerk_user_id`, NEVER `email`.
- **Trade-off:** lazy provisioning **cannot invite-gate** and cannot backfill email from the token
  (only `sub` is available). Reverting to invite-only later requires re-enabling the webhook **or**
  adding a Clerk custom JWT claim.
- **No new migration:** `users.username` is already `UNIQUE` + nullable (migration `0001`) and
  `clerk_user_id` is already `UNIQUE` (migration `0003`). Runtime gate: lazy upsert requires the
  `0003` columns (`clerk_user_id`/`is_admin`/`display_name`) to be live in Supabase before relying
  on it in prod (DONE — `0003` applied, Local=Remote).
- **Onboarding:** new users land on `/onboarding` (`<SignUp forceRedirectUrl="/onboarding">`,
  middleware-protected) to claim a username (`^[a-z0-9_]{3,30}$` + reserved blocklist), then
  redirect to `/blog/@username`. Execution detail in `00.tasks.md` Phase D D-lazy/D-onb.

**D4 Admin — findings + decisions (Cycle 3, 2026-06-26):**

- **Admin delete = SOFT-delete (`deleted_at`), acting on ANY post by id — NOT hard-delete.**
  Rationale: `posts.parent_id` is `ON DELETE CASCADE`, so a true `DELETE` cascades to child posts;
  and `post_tags`/`files`/`post_categories` lack FK cascade, so a true `DELETE` orphans their rows.
  Soft-delete (`deleted_at` stamp) is therefore the correct moderation primitive. Admin routes are
  NON-owner-scoped (operate by id under `requireAdmin`), unlike the owner-scoped author write paths.
- **Admin bootstrap gap + decision (Option A — manual SQL).** Lazy provisioning sets
  `is_admin=false` and there is **no grant UI**, so the **first admin is created manually**: after
  the owner signs up + completes onboarding, run Supabase SQL
  `UPDATE users SET is_admin=true WHERE username='<owner>'` (D4.0b); verify via
  `GET /v1/api/users/me` → `isAdmin:true`. **Match key:** lazy-provisioned rows have `email` NULL
  (only `clerk_user_id` stored at provision; `username` claimed at onboarding), so the bootstrap SQL
  MUST match by `username` (or `clerk_user_id`), **NEVER** by email. Owner = `taek.lim.us@gmail.com`
  (Clerk identity). **Future option (noted, not built):** an env allowlist `ADMIN_CLERK_IDS` checked
  in `resolveAuth` for self-bootstrap without manual SQL.
- **D4.2 `/admin/invites` — DEFERRED/DROPPED.** Under open self-signup there is **no enforcement
  point** (webhook soft-deleted; lazy provisioning ignores invites), so an invite-management UI
  would create a **false-authorization illusion** (codes that gate nothing). Reviving invite-gating
  requires EITHER re-enabling the Svix webhook (restore `x_clerk.ts`/`x_index.ts`, re-mount
  `/api/webhooks`, restore `express.raw`, re-register the Clerk endpoint + `CLERK_WEBHOOK_SECRET`)
  OR adding a Clerk custom JWT claim checked in `resolveAuth`. Recorded as a known trade; not built.

**Top risks:**
1. **IDOR / privilege escalation** — the `pg` Pool bypasses RLS, so a missing `WHERE user_id=$caller`
   has **no DB safety net**. Mitigation: a single `requireAuth` middleware as the sole source of
   `userId` + owner-scoped queries + jest tests proving user A cannot mutate user B's post.
2. **Deprecated-method trap** — the old JWT-template/`clerk_user_id()` approach; this rewrite fixes it.
3. **Unprovisioned user** — a Clerk user with no local `users` row would 500. Mitigation (now
   shipped): lazy provisioning in `resolveAuth` creates the row on first authenticated hit
   (idempotent, race-safe). *(Replaces the original "idempotent webhook + fail-soft 403".)*

### Phase E2 — Comment Edit + Comment Pagination

**Two features:**

1. **Comment EDIT** — `PATCH /v1/api/comments/:commentId`. Authorization = author OR post-owner OR admin (**identical to delete**, via a shared `assertCommentMutable`). Editing a soft-deleted (tombstoned) comment is rejected `409 CommentDeletedError`. Reuses `MAX_BODY_LENGTH=4000` trim/non-empty validation. **Key decisions:**

### Phase F — Blog Image Upload Rework (Express proxy + 2-step resize)

> **Intent:** replace the Phase D2 "signed-URL direct-to-Supabase from the browser" upload
> (Decision #14) with an **Express-proxied, server-resized** pipeline. The browser POSTs multipart
> to Express; Express resizes with `sharp` and uploads server-to-server to Supabase Storage. The Next
> SSR server is NOT involved; the browser never talks to Supabase directly. Owner + architect
> pre-decided; the design is FINAL. Atomic breakdown in `00.tasks.md` Phase F (F1–F17). SUPERSEDES
> Decision #14 → Decision #27.

**Why Option B (Express proxy):** keeps the Supabase secret server-side; lets the server validate and
normalize the actual bytes (security boundary); avoids needing Supabase bucket CORS. **Trade-off
accepted:** the Vercel serverless 4.5MB request-body limit — mitigated by a step-1 client resize.

**2-step resize:**
- **Step 1 (browser, transport-only — NOT a security boundary):** downscale long edge ~2000px → webp
  q~0.9 to land under the 4.5MB Vercel limit, via **native Canvas** (`createImageBitmap(file,
  { imageOrientation: 'from-image' })` for EXIF orientation → draw to canvas →
  `canvas.toBlob('image/webp', 0.9)`; if the blob still exceeds ~4MB, re-encode at lower quality).
  **NO library** — `browser-image-compression` rejected (unmaintained, last release 2023-03).
- **Step 2 (Express, authoritative + security):** `sharp(input,{failOn:'error'}).rotate().resize(1600,
  1600,{fit:'inside',withoutEnlargement:true}).webp({quality:80}).toBuffer()`. sharp decode failure =
  magic-byte validation (reject non-images); EXIF auto-stripped (no `.withMetadata()`); enforce a
  final-size cap.

**Endpoint:** `POST /v1/api/posts/images` (multipart `file`, `requireAuth`) → `200 { success:true,
data:{ publicUrl } }`. multer `memoryStorage`, route-level only, `limits.fileSize` ~5MB. Error mapping:
no file → 400 · multer `LIMIT_FILE_SIZE` → 413 · `not_an_image` → 415 · `too_large` → 413 · storage
unconfigured → 503 · upload fail → 502.

**Dependency versions (latest-stable, verified 2026-06-28):**
- Server (`apps/dashboard_server` `dependencies`): `sharp@^0.35.2` (Node 24 OK; on Vercel the
  `@img/sharp-libvips-linux-x64` binary MUST be bundled — add `vercel.json` `includeFiles` for the
  Express project), `multer@^2.2.0` (Express 5 compatible), `@types/multer` (devDependency).
- Client (`apps/dashboard_client_web`): **none** (native Canvas).
- Constraint: Vercel serverless request body limit = **4.5MB** (unchanged).

**Files (architecture already settled — no redesign):**
- Server new: `helpers/imagePath.ts` (`buildImagePath` + `ALLOWED_IMAGE_TYPES`, ext fixed `webp`, keep
  `${userId}/` prefix isolation — extracted from the old `signImageUpload`), `helpers/transformImage.ts`
  (pure buffer→buffer sharp transform; returns ok / `not_an_image` / `too_large`),
  `helpers/uploadImage.ts` (Supabase server-to-server `POST /storage/v1/object/{bucket}/{path}` with
  the secret key → publicUrl).
- Server edit: `controllers/posts/posts.write.ts` — remove the `/images/sign` route, add the `/images`

- Client new: `utils/resizeImage.ts` (`resizeImage(file: File): Promise<File>`, native Canvas),
  `apis/uploadImage.api.ts` (FormData POST; do NOT set `Content-Type` — let the browser set the

  `hooks/useImageUpload.hook.ts` internals to `resizeImage → uploadImage`, KEEPING the public

**Non-goals:** changing the blog_agent direct-write image path (it uploads to Supabase Storage
