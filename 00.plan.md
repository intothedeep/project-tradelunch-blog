## Product Identity

Personal portfolio + fintech dashboard for Taek Lim (GitHub: tradelunch).
The site serves two purposes under one Next.js 16 monorepo:

1. **Blog / portfolio** — curated technical posts, single-author, read-only. Posts are inserted via admin scripts; no authoring UI exists.
2. **Financial markets dashboard** — TradingView-style chart + watchlist with indicators, drawing tools, and real intraday intervals. Currently on mock data; Cycle 3 wires it to the real backend.

Domain: `my.prettylog.com`.

## Package Architecture (RESOLVED 2026-06-24)

Decision: "Option A + B" from architecture review — lean `@repo/types` package for API-boundary types; `@repo/markdown-parsing` retains shared logic.

- **`@repo/types`** — no build step; raw TS; `main`/`types` → `src/index.ts`. Holds API-boundary types shared between front and back: `ETreeNodeType`, `TCategoryTreeNode`, `TPostTreeNode`, `TCategoryTreeResponse`, `TTreeNode`, `TTreeNodeWithChildren`. Added `transpilePackages: ['@repo/types']` to `next.config` (package ships a runtime enum).
- **`@repo/markdown-parsing`** — retains shared parsing logic (TOC, snowflake, markdown file extraction). Re-exports the moved types for back-compat.
- Import sites in both apps were repointed to `@repo/types`. Four server publish-scripts that imported from `@repo/markdown-parsing/src/types` raw path now use the `/types` subpath.
- `@repo/assets` dead dependency removed from client; server's fragile `require.resolve("@repo/assets/...")` in `home.ts` replaced with `path.join`.
- Rationale: shared TYPE contract between front+back is valuable for API-boundary types only; kept separate from the markdown-parsing LOGIC package.

### Phase B — DB Migration: AWS → Supabase + Real Dashboard Data

**Goal:** Retire the AWS Postgres + SSH tunnel. Wire the financial dashboard to real market data served from Supabase via the backend.

**What "done" means:**
- `dashboard_server` reads all data from Supabase via `pg` Pool + raw SQL (no Sequelize, no SSH tunnel, no supabase-js).
- The Next.js app's Cycle 3 Server Actions (`getDashboardSnapshot.action.ts`, `getDashboardHistory.action.ts`) switch from mock to backend when `DASHBOARD_DATA_SOURCE=backend`.
- All 22 dashboard items show real values and 1-day change %.

**Data access (RESOLVED):** `pg` Pool (`max: 1`) against Supabase transaction pooler (port 6543). Raw SQL throughout. No supabase-js.

**Ingestion cron (DEFERRED — post-refactor):** No market-data ingestion cron is implemented in Phase B. The cron is future work, preferred implementation is a GitHub Actions scheduled workflow (avoids Vercel's 10s function timeout limit). Without an active cron, Supabase's 7-day auto-pause is an accepted known risk: fallback is a manual resume via Supabase dashboard, or a lightweight keepalive ping added later.

**Storage (DECIDED 2026-06-24):** Blog images migrate from AWS S3 to Supabase Storage (permanent free tier: 1 GB). AWS S3 SDK, `lib/awsS3.ts`, and the four AWS S3 env vars (`AWS_REGION`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_S3_BUCKET`) are retired as part of this phase. Until Phase B ships, the existing S3 code stays in place untouched.

**CDN_ASSET_POSTS (DECIDED 2026-06-24 — "CDN Option A"):** `CDN_ASSET_POSTS` will point at Supabase Storage for blog images. Later this can be fronted by a Cloudflare CNAME for a custom domain (CNAME a domain to the Supabase Storage public URL). The env var name and contract are unchanged regardless of whether the Cloudflare CNAME is active. `apps/dashboard_server/.env.example` comment for `CDN_ASSET_POSTS` should read: `# Supabase Storage public URL for blog images (later: Cloudflare CNAME to Supabase Storage)`.

**Scope:**

1. Supabase schema for market data:

2. Backend read endpoints (raw SQL via pg Pool):
   - `GET /api/dashboard/snapshot` — returns `IDashboardSnapshot`.

3. Next.js Cycle 3 wiring (Server Actions + Zod + polling hook) — tasks already in `00.tasks.md` Cycle 3 Groups A–B.

4. Retire AWS infra: SSH tunnel code removed; Sequelize removed; `pg` Pool reconfigured to point at Supabase pooler (port 6543) instead of AWS.

5. **Blog image storage migration — AWS S3 → Supabase Storage:**
   - Create `blog-images` bucket in Supabase Storage (public, permanent free tier).
   - Upload existing blog images from S3 to Supabase Storage bucket.
   - Update `dashboard_server` image-serve logic to read from Supabase Storage URL instead of S3.

   - Remove `aws-sdk` / `@aws-sdk/*` package from `dashboard_server` dependencies.

**Acceptance criteria:**

### Phase D — Multi-User Blog Platform (Invite-Only)

**Goal:** Turn the blog from a single-author read-only site into a multi-user platform where invited users can sign up, write, and publish their own articles. Signup is invite-only: new accounts require an admin-issued invite code or Clerk allowlist approval. Open public signup is explicitly out of scope.

**Signup model (RESOLVED):** Invite-only. An `invites` table in Supabase tracks invite codes (or Clerk's allowlist feature is used). No user can self-register without a valid invite. This removes the need for aggressive moderation at launch and keeps the platform controlled.

This is the largest phase by effort. It is broken into sub-milestones. Do not begin until Phase B is stable.

**What "done" means for the full phase:**
- A user with a valid invite code can sign up with Clerk (email + OAuth).
- Authenticated users can create, edit, and publish their own posts.
- Each user has a profile page at `/blog/@[username]`.
- A post is visible to other users only after the author publishes it.
- The original Taek Lim blog (`/blog/@taeklim`) continues to work unchanged.
- Moderation: an admin (Taek Lim) can unpublish or delete any post.
- No multi-tenancy billing; platform is free for invited users.

#### Sub-milestone D1 — Auth Foundation (Clerk + Supabase JWT bridge, Invite-Only)

**Scope:**
- Clerk project created; `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` and `CLERK_SECRET_KEY` set in Vercel.
- `@clerk/nextjs` installed in `dashboard_client_web`.
- Supabase Third-Party Auth configured: Clerk JWT template → Supabase JWT secret.
- SQL helper `clerk_user_id()` added to Supabase (extracts `sub` from JWT).
- `users` table in Supabase: `id` (matches Clerk `userId`), `username`, `display_name`, `created_at`.
- `invites` table in Supabase: `code` (unique), `used_by` (nullable), `created_at`. Admin inserts invite codes manually.
- Invite enforcement: sign-up flow requires a valid unused invite code; backend validates on `user.created` webhook before inserting into `users`; invalid code returns 403.
- Clerk webhook (`user.created`) → Supabase `users` insert via backend endpoint (only if invite valid).
- RLS: blog posts table gains `owner_id` column; SELECT policy allows `owner_id = clerk_user_id()` or `published = true`.
- Sign-in / sign-up flows via Clerk-hosted pages or embedded `<SignIn />` / `<SignUp />` components.
- Protected route middleware in Next.js: redirect unauthenticated users away from `/write`, `/settings`.

**Acceptance criteria:**
- Unauthenticated user can read all published posts.
- Authenticated user can reach `/write`; unauthenticated user is redirected to sign-in.
- Sign-up with an invalid invite code does not create a row in `users` table; response is 403.
- Sign-up with a valid unused invite code creates a row in `users` table within 5 seconds; invite `used_by` is set.
- `clerk_user_id()` returns the correct Clerk user ID in a Supabase RLS policy test.
- Sign-out clears the session; protected routes redirect correctly.
- Clerk MAU count is zero cost at zero users (free tier: 10,000 MAU).

**Non-goals for D1:**
- Writing or publishing UI.
- Admin moderation.
- Profile pages.
- Self-service invite request flow.

#### Sub-milestone D2 — Authoring UI (Editor + Draft Save)

**Scope:**
- `/write` route: Markdown editor (e.g. CodeMirror or plain `<textarea>` with preview pane).
- Draft auto-save to Supabase `posts` with `published = false` every 30 seconds or on blur.
- `POST /api/posts` backend endpoint — inserts new draft; owner set from Clerk JWT (`clerk_user_id()`).
- `PATCH /api/posts/:id` — updates title, body, tags; owner-only (RLS enforced).
- `DELETE /api/posts/:id` — soft-delete (`deleted_at` timestamp); owner-only.
- Post slug: auto-generated from title on first save; editable before publish.
- Image upload: author can attach images; stored in Supabase Storage (free: 1 GB).
- Tag field: free-form comma-separated; stored as `text[]` in Supabase.

**Acceptance criteria:**
- Author can create a new post; draft is saved without publishing; draft not visible to other users.
- Author can edit and re-save a draft; last-saved timestamp visible in editor UI.
- Author can delete a draft; row gets `deleted_at` set; post disappears from author's list.
- Image attached in editor appears in post preview.
- Unauthenticated `POST /api/posts` returns 401.
- Another authenticated user's `PATCH /api/posts/:id` on a post they do not own returns 403.

**Non-goals for D2:**
- Rich-text WYSIWYG (Markdown only).
- Collaborative editing.
- Scheduling / future-dated publish.

#### Sub-milestone D3 — Publish + Profile Pages

**Scope:**
- "Publish" button on editor: sets `published = true`; post becomes publicly readable.
- "Unpublish" button: reverts `published` to `false`.
- `/blog/@[username]` — user profile page listing that author's published posts (infinite scroll, same UX as existing `@taeklim` page).
- `/blog/@[username]/[slug]` — individual post page (existing Markdown renderer reused).
- Author byline on each post card (avatar from Clerk, display name from `users` table).
- `/settings/profile` — user can update `display_name` and bio.

**Acceptance criteria:**
- Published post appears on author's profile page within 5 seconds of clicking "Publish".
- Unpublished post is not accessible at its URL by any other user (returns 404 or 403).
- `/blog/@[username]` loads for any registered user; non-existent username returns 404.
- Author byline displays correct `display_name` and avatar.
- User can update `display_name` in settings; change reflected on their profile page without hard-reload.

**Non-goals for D3:**
- Comment system.
- Social follow / feed.
- Search.

#### Sub-milestone D4 — Admin Moderation

**Scope:**
- Admin role: `is_admin = true` column in `users` table; set manually in Supabase for Taek Lim.
- `/admin/posts` — admin-only page listing all posts (published + drafts) with unpublish and hard-delete actions.
- `/admin/invites` — admin-only page to generate and revoke invite codes.
- Backend endpoints: `PATCH /api/admin/posts/:id` (unpublish), `DELETE /api/admin/posts/:id` (hard-delete); admin-only via RLS + Clerk role check.
- `POST /api/admin/invites` — creates a new invite code row in `invites` table; admin-only.
- Moderation does not require any UI for end users; admin page is minimal.

**Acceptance criteria:**
- Non-admin authenticated user cannot access `/admin/posts` (redirect to home).
- Admin can unpublish any published post; post disappears from public listing within 5 seconds.
- Admin can hard-delete a post; row removed from `posts` table; URL returns 404.
- Non-admin `DELETE /api/admin/posts/:id` returns 403.
- Admin can generate an invite code via `/admin/invites`; code appears in `invites` table.

**Non-goals for D4:**
- Comment moderation.
- User banning.
- Audit log.
