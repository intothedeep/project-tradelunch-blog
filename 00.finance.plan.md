# 00.finance.plan.md — Financial Surface Split (App + DB)

> Scoped plan for splitting the **financial data surface** out of the blog/portfolio
> stack: separate deployment + a self-hosted **Oracle VM PostgreSQL 17** database,
> while blog/portfolio stays on **Vercel + Supabase**. This is a per-workstream doc
> (owner-approved deviation from the "three living docs" rule) because the split is a
> large, self-contained migration. Tasks: [`00.finance.tasks.md`](./00.finance.tasks.md).

## 1. Problem & intent

Vercel + Supabase billing spiked. The owner attributes it to the heavy **financial
surface** (backtest, dashboard, SEC filings, market-cap ranking, 13F funds, screener,
symbols, politician disclosures, market_history, weekly rankings, plus the Python
`apps/stock_collector` GitHub Actions crons).

Goal:

1. Keep **blog + portfolio as ONE app** on Vercel + Supabase — unchanged.
2. **Split the financial app** into its own deployment.
3. **Split the DB**: blog tables STAY on Supabase; **financial tables MOVE to a
   self-hosted Oracle Cloud VM running PostgreSQL 17** (Oracle = VM host; engine = plain PG17).
4. Repoint the financial **GitHub Actions** workflows to the Oracle DB.

## 2. ⚠️ Diagnosis — RESOLVED (2026-07-06)

**Answer: a crawler campaign, not steady-state load.** Usage screenshots confirmed BOTH
providers exploded from ONE body of traffic (Vercel Fast Origin Transfer 49.86GB ≈
Supabase egress 32GB), spiking 07-03..05. Over-limit meters: Vercel **Fluid CPU 22h/4h**,
**Fast Origin Transfer 49.86/10GB**, **Edge Request CPU 12h/1h**; Supabase **egress
32/5GB (27GB overage)**. No cron/deploy started ~07-02 → external scraper hitting
DB-backed finance routes (unbounded `funds.ts:195` / `dashboard.ts:140`), bypassing the
spoofable UA gate; **no `dashboard_client_web/middleware.ts` front-edge gate exists**.

**Consequence for this plan:** the migration fixes Supabase egress **fully** but does NOT
un-pause Vercel — Edge/Fluid CPU are front-side & crawler-driven. The real un-pause path
is **Phase 0-emergency** in the tasks doc (front-edge crawler gate = permanent fix).
The DB move proceeds as the deliberate structural fix, not under billing pressure.

The DB-move only fixes **one meter**. Original gate reasoning (kept for record):

| Cost                                            | Meter    | Oracle move fixes it? |
| ----------------------------------------------- | -------- | --------------------- |
| Supabase **database egress** (DB→Express bytes) | Supabase | ✅ Yes                |
| Vercel **bandwidth** (edge→browser bytes)       | Vercel   | ❌ No                 |
| Vercel **function GB-hours / invocations**      | Vercel   | ❌ No                 |
| Supabase **storage egress** (blog images)       | Supabase | ❌ No (blog, stays)   |

Key facts already established:

- The `stock_collector` crons are **write-dominant** (idempotent upserts = ingress;
  Supabase does not bill ingress) → the "heavy crons" are likely NOT the egress driver.
- Cache headers **exist** on all financial routes (`s-maxage` + `stale-while-revalidate`).
  Whether Vercel CDN actually HITs them is unverified (prod is paused).
- **Two unbounded queries confirmed in code** (likely egress amplifiers):
    - `dashboard_server/src/controllers/funds/funds.ts:195` — holdings `ORDER BY h.value_usd DESC`, **no LIMIT**.
    - `dashboard_server/src/controllers/dashboard/dashboard.ts:140` — `market_snapshots ... ORDER BY category, seq`, **no LIMIT**.

**Gate task (Phase 0):** owner pulls Vercel Usage (which meter tripped the pause) +
Supabase Usage (DB egress vs storage egress) + CDN HIT rate once prod is un-paused.
The owner has chosen Oracle self-host as the target **regardless**, since separating the
financial DB's egress from the blog's Supabase quota has standalone value — but any
**Vercel** function/bandwidth spike must be fixed separately (it is not solved by the DB move).

## 3. Decisions

### D1 — Repo: STAY monorepo (no new git repo)

Both the architect and advisor concur. Egress is a DB/network problem, not a
source-control one. Financial and blog are already separate Vercel deployables inside
one Turborepo. Blog↔financial DB coupling is **zero** (no FK / no JOIN across the two
table sets — verified). Decouple at the **deployment + connection-string** boundary,
keep one repo, one history, shared `@repo/*` (only `types` + config packages are shared).

### D2 — DB target: Oracle VM PostgreSQL 17 (self-host) — owner decision

Owner already holds an Oracle instance and wants full control + 10TB/mo free egress.
Advisor's cheaper managed-PG alternative (Neon/2nd Supabase) is **noted but declined**.
Because self-host trades a cost problem for an ops problem, the following are **required
tasks, not optional**: server-side pooler (PgBouncer/pgcat), automated off-box backups +
WAL archiving, a **tested restore**, TLS-only public exposure, and uptime monitoring.

### D3 — Topology: STANDALONE finance app (REVISED 2026-07-07, owner decision)

Supersedes the earlier "carve API only, keep finance routes in the blog Next app."
Finance becomes a **fully standalone application** — its own frontend + API + DB — that
deploys independently. THREE new workspaces in the SAME monorepo:
- `apps/finance_web` — NEW Next 16 app; the existing finance UI is MOVED here (not
  rebuilt), with its OWN `layout.tsx`, `proxy.ts`, Clerk, globals. Own Vercel project.
- `apps/finance_api` — Express, carved from `dashboard_server` financial controllers,
  runs **on the Oracle VM next to PG17** (localhost reads → egress never crosses a
  metered boundary). Single systemd/PM2 process behind Caddy/nginx TLS (Option A).
- `packages/finance-migrations` — financial DDL SSOT for Oracle PG17.

`finance_web` fetchers point at `FINANCE_API_BASE_URL` (Oracle `finance_api`). Shared
`@repo/{types,*-config}` stay via `workspace:*`. Clean boundaries → cheap later
extraction to a separate repo. Auth: finance_web gates ALL routes (owner-only tooling);
the emergency `proxy.ts` gate MOVES here. Clerk = shared instance (Option A, KISS).
`components/ui` atoms: finance uses only `table` → duplicate (no `@repo/ui` today).
Full carve inventory + phased sequence in `00.finance.tasks.md` (P1–P11).

## 5. DB split

### 5.1 Table inventory (SSOT = `apps/dashboard_server/supabase/migrations/0001–0034`)

**STAY on Supabase (blog/portfolio):** `users`, `posts`, `categories`, `files`, `tags`,
`post_tags`, `post_categories`, `invites`, `post_favorites`, `post_likes`, `comments`.

**MOVE to Oracle PG17 (financial):** `market_snapshots`, `market_history`,
`tracked_symbols`, `market_rankings`, `symbol_fundamentals`, `batch_log`,
`sec_filings`, `sec_holdings`, `security_map`, `fund_registry`, `politician_registry`,
`politician_trades`, `politician_committees`, `committee_sector_map`, `signal_backtest`,
`gex_daily`, `mv_sec_new_positions` (MV). **Classify 0032–0034 objects at cut time.**

**DUPLICATE (per-DB, NOT moved):** `error_log` — the blog also writes to it (SSR/client
errors → dashboard_server `/api/error-logs`), so it STAYS on Supabase for the blog AND the
finance DDL creates its OWN `error_log` on Oracle for finance_api's error handler. Same table
name, two independent DBs. `0001_finance_init.sql` MUST include `error_log`. (Q2 resolved 2026-07-08.)

### 5.2 DDL generation (fresh schema on Oracle)

Migrations interleave blog + financial by number → do NOT hand-replay by number.
Generate financial-only DDL from the live schema and commit as the new SSOT:
`pg_dump --schema-only --no-owner --no-privileges -t <financial tables> $POSTGRES_URL_NON_POOLING`
→ `packages/finance-migrations/0001_finance_init.sql`. Strip Supabase-isms
(`auth.`/`storage.`/RLS/`supabase_*` roles). Apply with `psql -v ON_ERROR_STOP=1`, then
`REFRESH MATERIALIZED VIEW mv_sec_new_positions;`. New financial migrations land here
going forward; blog migrations stay in `dashboard_server/supabase/migrations/`.

### 5.3 Cross-DB references

None to worry about. Financial tables do not FK into blog tables. **`error_log` /
`batch_log`** are the only shared-ish ones — they follow the financial side (the
collector + finance runtime write them); blog client-error ingest then also logs to
Oracle (acceptable; it's diagnostics). Confirmed as an open item (Q2).

### 5.4 Env / connection

- `finance_api`: reads **`POSTGRES_URL` / `POSTGRES_URL_NON_POOLING`** (its own `env.schema.ts`;
  `DATABASE_URL = env.POSTGRES_URL`) → point at Oracle localhost `:5432` (or `:6432` via PgBouncer).
  NB the earlier `FINANCE_POSTGRES_URL`/`FINANCE_DATABASE_URL` naming here was STALE — the code
  never read it; setting it at deploy leaves the pool pointing at nothing (verified 2026-07-08).
- `stock_collector`: `DATABASE_URL` → Oracle **session/direct** (`:5432`, never a txn
  pooler — it uses `prepare_threshold=None`; keeps `POSTGRES_URL_NON_POOLING` fallback).
- `dashboard_server` (blog): keeps `POSTGRES_URL` → Supabase; drops all financial DB access.
- `dashboard_client_web`: add `FINANCIAL_API_BASE_URL`; financial fetchers use it.
