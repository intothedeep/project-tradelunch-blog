## 1. 병합 목표 & 가치

### 왜 합치는가

| 현재 상태 | 병합 후 |
|---|---|
| 두 저장소가 동일한 Postgres 스키마를 각자 정의 | 스키마 SSOT 단일화, drift 감지 자동화 |
| S3 버킷/CDN 경로 설정이 두 곳에 중복 | 환경 변수 한 곳에서 관리 |

| Python 에이전트와 TS 서버의 릴리즈가 비동기 | turbo에서 함께 lint/test 선택 실행 가능 |
| 협업자 온보딩 시 두 레포 클론 필요 | 클론 1회로 전체 발행 파이프라인 접근 |

### 핵심 가치 (YAGNI 준수)
1. **DB 스키마 드리프트 제거** — `apps/dashboard_server/schema/tradelunch.schema.sql`(SSOT)와 `apps/blog_agent/schema/tradelunch.schema.sql`(복제본) 간 자동 diff
2. **단일 저장소에서 발행 파이프라인 실행** — `python cli_multi_agent.py`가 모노레포 안에서 동작
3. **환경 변수 가시성 확보** — 두 app의 `.env.example`을 같은 레포에서 비교 관리

## 2. Phase 구분 (확정)

이 병합은 두 Phase로 명확히 분리한다. **이번 세션(Phase A)과 다른 세션(Phase B)은 절대 혼합하지 않는다.**

### Phase A — Lift-and-Shift 병합 (이번 세션)

**목표:** 코드 로직 무변경 상태로 `apps/blog_agent/`에 이전 + turbo 통합 + 스키마 드리프트 감지.

**원칙:**
- 런타임 인프라 미접촉 → 롤백 안전
- Supabase/Vercel 자격증명 불필요
- 기존 `DB_PG_*`/`AWS_*` env 계약 유지
- 코드 diff는 파일 이동 + `package.json` 추가 + `turbo.json` 수정 + `x_` 이름변경만

**수용 기준:**
- `apps/blog_agent/`에 코드 이전 완료, 원본 레포 미변경
- `pnpm --filter blog_agent test` 실행 시 pytest 호출됨 (deps 설치 환경에서 통과)
- `turbo run build`와 `turbo run dev`에 blog_agent 미개입
- `pnpm check:schema` exit 0 (두 스키마 파일 동일)
- 코드 로직 무변경 검증: diff가 이동 + 설정 파일 + x_ 이름변경만 포함

### Phase B — Supabase/Cloudflare Repoint (별도 세션, `00.migration.md` S8 연계)

**전제 조건 (USER 선행 작업 필요):**
- Supabase Storage `blog-images` 버킷 생성
- Cloudflare CNAME `CDN_ASSET_POSTS` 도메인 설정
- Supabase pooler `DATABASE_URL` 발급 확인

**목표:**
- DB 연결: `DB_PG_*` → `DATABASE_URL`(asyncpg + Supabase pooler) 전환
- 스토리지: boto3/AWS S3 → Supabase Storage S3 호환 엔드포인트(`endpoint_url`) + `blog-images` 버킷
- CDN: `CDN_ASSET_POSTS` → Cloudflare CNAME 재지정
- RDS CA cert / SSH 터널 / boto3-AWS 잔재 정리

**이번 세션에서 절대 진행하지 않음.**

## 4. 스키마 SSOT 정책 (Opt-3 확정)

**SSOT:** `apps/dashboard_server/schema/tradelunch.schema.sql`

**이유:**
- `supabase/migrations/`가 여기서 파생됨
- `config.toml`의 `schema_paths=[]`로 supabase CLI가 `./supabase/migrations`만 인식 → 스키마를 `packages/`로 이동 불가
- Docker/CI 비호환 및 비런타임 사용으로 심링크/공유패키지 추출은 과설계

**복제본:** `apps/blog_agent/schema/tradelunch.schema.sql`
- agent는 DDL을 런타임 실행하지 않음 (SQLAlchemy 모델 설계 참조용)
- 바이트 동일성을 `pnpm check:schema`로 자동 감지

**드리프트 방지:**
```
pnpm check:schema
# 내부: diff apps/dashboard_server/schema/tradelunch.schema.sql apps/blog_agent/schema/tradelunch.schema.sql
# exit 0이면 동일, exit 1이면 drift 감지
```

## 5. Turborepo 통합 정책 (확정)

**접근법:** 얇은 `package.json` 스텁 + `uv run` 위임

```json
// apps/blog_agent/package.json (핵심)
{
  "name": "blog_agent",
  "scripts": {
    "start": "uv run python cli_multi_agent.py",
    "test": "uv run pytest",
    "lint": "uv run ruff check .",
    "check-types": "uv run mypy ."
  }
}
```

**규칙:**
- `start` 스크립트는 `turbo run dev` 그래프에서 **제외** (dev/build 파이프라인 미개입)
- `test`, `lint`, `check-types`만 `--filter=blog_agent`로 선택 실행
- Python `.venv`는 `apps/blog_agent/.venv/` 로컬 유지
- Node 의존성 없음 (turbo 인식용 스텁만)

## 7. 리스크 & 의존성

| 리스크 | 심각도 | 대응 |
|---|---|---|
| `uv` 네트워크 제약으로 deps 설치 불가 | 중간 | 구조 작업은 완료, 런타임 테스트 보류 허용 |
| `configs/paths.py` 절대경로 mkdir 실패 | 낮음 | Phase A에서 경로 수정 없음. 코드 로직 무변경 원칙에 따라 Phase B에서 처리 |
| `check:schema` diff가 BOM/줄바꿈 차이로 오탐 | 낮음 | `diff -b` 옵션 또는 md5 비교로 대응 |
| finance MVP 브랜치(`feature/finance`)와 충돌 | 낮음 | `apps/blog_agent/`는 기존 파일과 겹치지 않음 |
| Phase B 선행 작업 지연 | 낮음 | Phase A는 완전 독립 실행 가능 |

**외부 의존성 (Phase B 전용):**
- USER: Supabase Storage `blog-images` 버킷 생성
- USER: Cloudflare CNAME `CDN_ASSET_POSTS` 설정
- `project_tradelunch_agent_blog` 레포 접근 권한 (Phase A rsync 시)
