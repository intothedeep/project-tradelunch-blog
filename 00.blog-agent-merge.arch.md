## ADR-01. Turborepo Python 인식 방법

**결정:** 얇은 `package.json` 스텁을 `apps/blog_agent/`에 추가하고, scripts가 `uv run`으로 위임한다.

**근거:**
- turbo는 `package.json`이 있는 workspace를 인식한다. Node 의존성은 전혀 없어도 된다.
- `start` 스크립트를 turbo `dev` 파이프라인에서 제외함으로써 `turbo run dev`가 Python 프로세스를 시작하지 않는다.
- `test`, `lint`, `check-types`는 `--filter=blog_agent`로 선택 실행 가능하여 CI 통합이 용이하다.
- Makefile 대안(옵션 B)은 turbo 그래프 밖으로 나가 캐싱/병렬 실행 이점을 잃는다.

**거부된 대안:**
- Makefile/shell 스크립트만으로 실행 — turbo 그래프 밖이라 캐싱 불가, 온보딩 진입점이 분리됨
- turbo custom executor 플러그인 — 과설계, YAGNI 위반

**스텁 핵심 구조:**
```json
{
  "name": "blog_agent",
  "private": true,
  "scripts": {
    "start": "uv run python cli_multi_agent.py",
    "test": "uv run pytest",
    "lint": "uv run ruff check .",
    "check-types": "uv run mypy ."
  }
}
```

**turbo.json 변경 범위:** `test` 태스크에만 `blog_agent` 추가. `build`/`dev` 파이프라인 불변.

## ADR-02. 스키마 SSOT 소유권

**결정 (Opt-3):** `apps/dashboard_server/schema/tradelunch.schema.sql`이 authoritative. `apps/blog_agent/schema/tradelunch.schema.sql`은 복제본 유지.

**근거:**
- `supabase/migrations/`는 `config.toml`의 `schema_paths=[]` 설정으로 `./supabase/migrations`만 인식한다. 따라서 스키마를 `packages/db-schema/`로 추출하면 supabase CLI가 인식 불가 → Docker/CI 파이프라인 깨짐.
- blog_agent는 DDL을 런타임에 실행하지 않고 SQLAlchemy 모델 설계 참조용으로만 사용한다. 따라서 복제본의 위험도가 낮다.
- 심링크: Docker 빌드 컨텍스트에서 심링크는 예측 불가 동작을 유발한다. 거부.
- 자동 drift 감지(`pnpm check:schema`)로 복제본 이탈을 조기 발견한다.

**거부된 대안:**
- `packages/db-schema/` 신규 패키지 — supabase CLI `schema_paths` 경로 구속으로 실현 불가
- 심링크 — Docker/CI 비호환
- `supabase/migrations/`를 SSOT로 지정 — migration 파일은 증분 변경사항이지 전체 스키마 정의가 아님

**drift 감지 스크립트:**
```sh
# package.json scripts.check:schema
diff apps/dashboard_server/schema/tradelunch.schema.sql \
     apps/blog_agent/schema/tradelunch.schema.sql
```
exit 0 → 동일, exit 1 → drift 감지.

## ADR-03. Python 가상환경 위치

**결정:** `apps/blog_agent/.venv/` 로컬 유지.

**근거:**
- `.claude/rules/python.md`에 "isolated .venv per project" 명시.
- `apps/blog_agent/`를 독립 Python 프로젝트로 간주하면 규칙과 일치한다.
- 레포 루트 `.venv/`는 TS 툴체인과 혼재하여 격리 원칙 위반.
- `.gitignore`에 `apps/blog_agent/.venv/` 추가로 커밋 방지.

**거부된 대안:**
- 레포 루트 `.venv/` — TS/Python 격리 원칙 위반, python.md 규칙 위반

## ADR-04. 공유 환경 변수 계층

**결정 (Phase A):** 각 app 별 `.env` 유지. 공유 변수 통합은 Phase B에서 수행.

**근거:**
- Phase A는 lift-and-shift. 기존 `DB_PG_*`/`AWS_*` env 계약을 그대로 유지해야 롤백 안전하다.
- 레포 루트 `.env` 도입은 각 app의 `dotenv` 로드 경로 수정을 요구하며 코드 변경이 발생한다 → Phase A 원칙 위반.
- `apps/blog_agent/.env.example`을 레포에 추가하여 필요 변수 목록 가시화는 코드 변경 없이 가능.

**Phase B 결정 예정:**
- `DATABASE_URL` 단일 변수로 통합 (Supabase pooler)
- `AWS_*` 제거, Supabase Storage 자격증명으로 교체

**거부된 대안 (Phase A에서):**
- 레포 루트 `.env` 즉시 도입 — 코드 수정 유발, Phase A 원칙 위반

## ADR-05. CI 파이프라인 포함 여부

**결정 (Phase A):** CI 추가 없음. 로컬 `pnpm --filter blog_agent test` 검증으로 충분.

**근거:**
- Phase A 목표는 구조 이전이지 CI 확장이 아니다.
- `uv` 네트워크 제약 시 런타임 테스트 보류를 허용한다 (구조 작업은 완료 기준).
- GitHub Actions에 `uv` 설치 스텝 추가는 별도 작업으로 Phase B 이후 검토.

**거부된 대안:**
- Phase A에서 CI 스텝 즉시 추가 — 범위 초과, YAGNI 위반

## ADR-06. pyproject.toml 빌드 백엔드

**결정:** setuptools 유지. `packages.find`의 `"config"` → `"configs"` 오타만 수정.

**근거:**
- 기존 환경에서 검증된 빌드 백엔드를 교체하면 의존성 해석 위험이 발생한다.
- Phase A는 코드 로직 무변경 원칙. 빌드 백엔드 교체는 코드 변경이 아니지만 런타임 영향이 있다.
- 오타 수정은 기존 코드가 올바르게 동작하기 위한 최소 수정이므로 허용.

## ADR-07. `src/`, `temp/` 디렉터리 처리

**결정:** `x_src/`, `x_temp/`로 soft-delete (`.claude/CLAUDE.md` "rm -rf" 규칙 준수).

**근거:**
- 규칙: "never delete files directly. Rename with `x_` prefix so the user can verify and remove manually."
- `src/`는 미사용 아카이브, `temp/`는 임시 파일 — 이전 대상이 아님.
- USER가 확인 후 수동 삭제 가능.

## ADR-08. Phase A/B 분리 원칙

**결정:** Phase A(이번 세션)와 Phase B(별도 세션)는 하나의 PR/브랜치에 혼합하지 않는다.

**근거:**
- Phase B는 USER 선행 작업(Supabase 버킷, Cloudflare CNAME) 완료 후에만 실행 가능하다.
- Phase A PR이 Phase B 작업을 포함하면 검증 불가 코드가 main에 병합되는 위험이 있다.
- `00.migration.md` S8과 연계되는 Phase B는 별도 브랜치/PR로 추적한다.
